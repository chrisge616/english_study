# REBUILD AND SMOKE — English Study System (Gen2 Core v0.1)

This document defines the current **fresh rebuild** and **smoke verification** procedure for Gen2 Core.

Its purpose is practical:

- rebuild a clean local baseline
    
- verify the current frozen Gen2 behavior
    
- distinguish real regressions from expected stabilization gaps
    

This is the current operational validation path for Gen2.

---

## 1. Scope

This document is for:

- rebuilding the local SQLite state from scratch
    
- replaying the known smoke sample logs
    
- checking expected spot results
    
- checking the current automated test status
    

It is **not** a full test strategy for future generations.

---

## 2. Current Validation Philosophy

At the current stabilization stage, Gen2 is validated in two layers:

### Layer A — Smoke workflow

This is the main operational baseline.

It verifies that:

- the database can be initialized
    
- known sample daily logs can be ingested
    
- the known sample review log can be ingested
    
- expected representative word states are produced
    
- output files can be generated by the CLI
    

### Layer B — Automated tests

The repository also contains `tests/`, but the test layer is currently smaller than the operational baseline.

This means:

- smoke workflow is the primary source of confidence
    
- automated tests are still being aligned to the frozen Gen2 baseline

## 5. Fresh Rebuild Procedure

### Step 1 — Delete the existing SQLite database

```powershell
Remove-Item ".\data\study.db"
```

If the file does not exist yet, that is fine.

### Step 2 — Recreate the database

```powershell
study init-db
```

Expected result:

```text
[study] database initialized
```

### Step 3 — Re-ingest the daily smoke samples

```powershell
study ingest daily ".\logs\daily\2026-04-07_gen2.md"
study ingest daily ".\logs\daily\2026-04-08_gen2.md"
study ingest daily ".\logs\daily\2026-04-10_gen2.md"
```

Expected result shape:

- each command should return a dictionary-like summary
    
- `processed` should be greater than 0 on first ingest
    
- `skipped` should be `False` on first ingest
    

### Step 4 — Re-ingest the review smoke sample

```powershell
study ingest review ".\logs\review\2026-04-10_review.md"
```

Expected result shape:

- `processed` should be greater than 0 on first ingest
    
- `skipped` should be `False` on first ingest
    

### Step 5 — Run the key inspection checks

```powershell
study inspect word nuance
study inspect word advance
study inspect word reconstitute
```

Current expected spot results:

- `nuance` -> `WEAK`
    
- `advance` -> `STABLE`
    
- `reconstitute` -> `WEAK`
    

For `reconstitute`, the important meaning is:

- partial review did **not** collapse to full wrong
    
- slight progress was retained

## 7. Idempotent Ingest Check

The current Gen2 baseline includes **idempotent ingest** by `session_id`.

To verify this, re-run one of the already ingested files.

Example:

```powershell
study ingest daily ".\logs\daily\2026-04-07_gen2.md"
```

Expected result shape:

- `processed: 0`
    
- `skipped: True`
    
- `reason: session already exists`
    

This is correct behavior.

## 8. What Counts as a Passing Smoke Rebuild

A rebuild should be treated as passing when all of the following are true:

1. `study init-db` succeeds
    
2. the 3 daily sample logs ingest successfully on first run
    
3. the review sample log ingests successfully on first run
    
4. `nuance`, `advance`, and `reconstitute` match the expected baseline states
    
5. `study run daily` generates `output/daily_prompt.md`
    
6. `study run review` generates the 3 review output files
    
7. re-ingesting the same file is skipped as expected
    

If all 7 conditions hold, Gen2 should be treated as operationally intact.

## 9. Current Frozen Baseline Behaviors

The rebuild should be interpreted against the current frozen Gen2 policy, not against older assumptions.

Current expected behaviors:

- `wrong -> WEAK`
    
- `correct -> STABLE`
    
- `partial -> WEAK but with slight progress`
    
- scheduler may keep a small amount of risky `WEAK` reinforcement
    
- recently reviewed `STABLE` items should not come back immediately
    
- repeated ingest of the same session should be skipped
    

Frozen modules:

- `domain/policy.py`
    
- `domain/scheduler.py`
    
- `services/ingest_service.py`
    
- `storage/session_repo.py`
    

During stabilization, these should be treated as the current behavioral source of truth.

## 12. Troubleshooting During Rebuild

### Problem 1 — A daily ingest is skipped immediately

Check whether the database was actually deleted before rebuild.

Expected cause:

- old `study.db` still exists
    
- that session was already ingested earlier
    

### Problem 2 — Review ingest processed 0 items

Check:

- the review file is the completed result file, not just the prompt
    
- the machine block still exists
    
- the machine block JSON is valid
    
- fallback status table format is still intact
    

### Problem 3 — Expected spot words do not match

Check in this order:

1. correct smoke files were used
    
2. files were ingested in the intended order
    
3. no duplicate ingest confusion happened
    
4. you are reading the current word state, not an older cached assumption
    

### Problem 4 — Output files are missing

Check:

- the correct `run` command was used
    
- runtime folders exist
    
- the CLI starts successfully
    

### Problem 5 — `pytest` fails

Check whether the failure is:

- a true operational regression
    
- or a test that still reflects an older expectation
    

Use the smoke rebuild result as the first tie-breaker.

---

## 13. Decision Rule: Is It a Real Regression?

Treat it as a likely real regression only when:

- smoke sample ingest fails unexpectedly, or
    
- output generation fails, or
    
- expected spot states no longer match, or
    
- idempotent ingest no longer works
    

Treat it as a likely **stabilization/documentation/test-alignment issue** when:

- smoke rebuild passes
    
- operational outputs are correct
    
- but a legacy or narrow automated test still fails
